## 矩阵乘法



### mma

**1. MMA 指令代码实现（PTX 嵌入）**

在 C++ 中，我们通常使用 `asm` 块来调用底层的 MMA 指令。以下是一个典型的 `m16n8k16`（A 矩阵 16x16，B 矩阵 16x8）的片段：

```c++
// 假设我们使用 FP16 输入和 FP32 累加
// d = a * b + c
__device__ __forceinline__ void mma_m16n8k16_fp16_fp32(
    float* d,   // 结果寄存器 (8个 float)
    uint32_t* a, // A矩阵分片 (4个 uint32，每个含2个 fp16)
    uint32_t* b, // B矩阵分片 (2个 uint32，每个含2个 fp16)
    float* c    // 累加器输入 (8个 float)
) {
    asm volatile(
        "mma.sync.aligned.m16n8k16.row.col.f32.f16.f16.f32 "
        "{%0, %1, %2, %3, %4, %5, %6, %7}, " // 输出 D
        "{%8, %9, %10, %11}, "               // 输入 A
        "{%12, %13}, "                       // 输入 B
        "{%14, %15, %16, %17, %18, %19, %20, %21};" // 输入 C
        : "=f"(d[0]), "=f"(d[1]), "=f"(d[2]), "=f"(d[3]), "=f"(d[4]), "=f"(d[5]), "=f"(d[6]), "=f"(d[7])
        : "r"(a[0]), "r"(a[1]), "r"(a[2]), "r"(a[3]),
          "r"(b[0]), "r"(b[1]),
          "f"(c[0]), "f"(c[1]), "f"(c[2]), "f"(c[3]), "f"(c[4]), "f"(c[5]), "f"(c[6]), "f"(c[7])
    );
}

```

---

**2. 深度原理解析**

**A. 什么是 m16n8k16？**

这是 MMA 指令的**形状（Shape）**描述：

* **M=16, N=8**: 最终计算出的结果矩阵分片大小。
* **K=16**: 乘法运算时的收缩维（Dot Product 的长度）。
* 这意味着这一条指令就在硬件层面完成了  的半精度乘法累加。

**B. 数据映射 (Thread Mapping)**

这是 MMA 最硬核也最难的地方。在底层 MMA 指令中，数据不是连续存放在一个线程里的，而是**散布在 Warp 的 32 个线程中**：

* **A 矩阵**: 每个线程持有 A 矩阵的一部分寄存器。
* **指令对齐**: `mma.sync.aligned` 要求整个 Warp 必须严格同步，因为硬件执行时会从这 32 个线程的寄存器堆中并行抽取数据进入 Tensor Core 单元。

**C. 指令修饰符**

* `.row.col`: 指定了 A 矩阵是行优先（Row-major），B 矩阵是列优先（Col-major）。这直接影响硬件如何解析寄存器里的数据位。
* `.f32.f16.f16.f32`: 定义了混合精度路径。中间两个 `f16` 是输入，首尾两个 `f32` 是累加器和输出。

---

**3. MMA 与 WMMA 的本质区别**

| 维度 | WMMA (高层) | MMA (底层/PTX) |
| --- | --- | --- |
| **控制权** | 编译器决定寄存器分配 | 程序员手动分配寄存器 |
| **灵活性** | 受限于 C++ 接口定义的形状 | 支持更多硬件原生的形状（如极细长的矩阵） |
| **性能瓶颈** | 编译器可能插入多余的移动指令 | 可以通过手动优化寄存器复用达到极致性能 |
| **可见性** | 抽象了数据布局 | 必须显式处理每个线程负责的矩阵索引 |

---

**4. 为什么要用这么低级的指令？**

通常只有在编写像 **FlashAttention** 这种对性能压榨到极致的内核时，才会直接写 `asm mma`。因为它允许你绕过编译器的限制，直接控制寄存器的生命周期，从而显著降低 **Register Pressure**（寄存器压力），让 GPU 能跑更多的并发线程（Occupancy）。





### wmma

在 CUDA 编程中，**WMMA (Warp Matrix Multiply-Accumulate)** 是利用 NVIDIA Tensor Cores（张量核心）进行硬件级矩阵运算加速的关键技术。它自 Volta 架构（如 V100）引入，能在极短的时间内完成低精度（如 FP16）的矩阵乘加运算。

下面是一个使用 WMMA API 实现 $C = A \times B + C$ 的 C++ 示例代码，并附带详细的技术拆解。

------

**1. WMMA 矩阵乘法示例代码**

```c++
#include <mma.h>
#include <cuda_fp16.h>

using namespace nvcuda;

// 定义矩阵维度 (必须符合硬件限制，如 16x16x16)
const int WMMA_M = 16;
const int WMMA_N = 16;
const int WMMA_K = 16;

template<const int WMMA_M=16, const int WMMA_N=16, const int WMMA_K=16>
__global__ void hgemm_wmma_m16n16k16_naive_kernel(half* A, half* B, half* C, 
                                                  int M, int N, int K) {
  const int NUM_K_TILES = div_ceil(K, WMMA_K);
  const int load_gmem_a_m = blockIdx.y * WMMA_M;
  const int load_gmem_b_n = blockIdx.x * WMMA_N;
  if (load_gmem_a_m >= M && load_gmem_b_n >= N) return;
  
  wmma::fragment<wmma::accumulator, WMMA_M, WMMA_N, WMMA_K, half> C_frag;
  wmma::fill_fragment(C_frag, 0.0);
  
  #pragma unroll
  for (int k = 0; k < NUM_K_TILES; ++k) {
    wmma::fragment<wmma::matrix_a, WMMA_M, WMMA_N, WMMA_K, half, wmma::row_major> A_frag;
    wmma::fragment<wmma::matrix_b, WMMA_M, WMMA_N, WMMA_K, half, wmma::row_major> B_frag;
    
    wmma::load_matrix_sync(A_frag, A + load_gmem_a_m * K + k * WMMA_K,    K);
    wmma::load_matrix_sync(B_frag, B + (k * WMMA_K) * N  + load_gmem_b_n, N);

    wmma::mma_sync(C_frag, A_frag, B_frag, C_frag);
    
    __syncthreads(); 
  }
  wmma::store_matrix_sync(C + load_gmem_a_m * N + load_gmem_b_n, C_frag, N, 
                          wmma::mem_row_major);
}
```

------

**2. 核心概念与代码详解**

A. 什么是 Fragment (片段)？

在普通的 CUDA 核函数中，我们处理的是单个浮点数。而在 WMMA 中，数据的基本单位是 **Fragment**。

- `wmma::fragment` 是一个类模板，它将矩阵的一小部分分布存储在整个 Warp（32个线程）的寄存器中。
- **注意**：你不能直接像访问数组一样访问 Fragment 里的元素，必须通过 WMMA 提供的 API 进行操作。

B. 核心步骤拆解

1. **`fill_fragment`**: 相当于显存初始化 `memset`，通常用于将累加器（Accumulator）清零。
2. **`load_matrix_sync`**: 从显存（Global 或 Shared Memory）中加载矩阵数据。
   - `lda` (Leading Dimension): 表示矩阵主维度的跨度。如果是行优先，`lda` 就是矩阵的列数。
   - **Sync 含义**: 这是一个 Warp 级别的同步操作，意味着 Warp 内的所有 32 个线程必须同时到达这里才能完成加载。
3. **`mma_sync`**: 真正触发 **Tensor Core** 的指令。
   - 它在硬件层面执行 $D = A \times B + C$。
   - 这比用普通 CUDA Core 写的循环快得多，因为它是在专门的硬件电路上并行完成的。
4. **`store_matrix_sync`**: 将计算结果从寄存器写回到显存。

------

**3. 使用 WMMA 的硬性限制**

Tensor Core 虽然强大，但非常“挑剔”：

| **限制维度** | **要求**                                                     |
| ------------ | ------------------------------------------------------------ |
| **数据类型** | 输入通常为 `half` (FP16)，累加器可以是 `float` (FP32) 或 `half`。 |
| **矩阵尺寸** | 必须是预定义的固定尺寸。最常用的是 **16x16x16** (M-N-K)。    |
| **对齐要求** | 内存地址必须保持 **128-bit 对齐**（即 16 字节对齐），否则性能会大幅下降或报错。 |
| **执行单元** | WMMA 是 **Warp 级别**的操作。这意味着你必须确保 Warp 里的 32 个线程协同工作。 |

------

**4. 总结与进阶建议**

上面的代码只是一个“原子级”的演示。在实际的工程（如深度学习算子）中，通常需要：

1. **分块 (Tiling)**：将大矩阵拆分成多个 16x16 的小块。
2. **共享内存 (Shared Memory)**：先将数据从 Global Memory 搬运到 Shared Memory，再由 WMMA 从 Shared Memory 加载，以减少带宽瓶颈。
3. **双缓冲 (Double Buffering)**：利用异步拷贝（Async Copy）隐藏内存延迟。



### cublas

在 CUDA 编程中，使用 **cuBLAS** 库进行矩阵乘法（GEMM）是最高效的选择之一。cuBLAS 是 NVIDIA 提供的线性代数标准库，经过了高度优化。

---

**1. cuBLAS 矩阵乘法核心代码**

以下是一个完整的 C++ 示例，展示了如何初始化 cuBLAS 并执行矩阵乘法。

```cpp

#include <cuda_runtime.h>
#include <cublas_v2.h>

      exit(EXIT_FAILURE);
    }
  }
}

  if (g_handle != nullptr) {
    cublasStatus_t status = cublasDestroy(g_handle);
    if (status != CUBLAS_STATUS_SUCCESS) {
      printf("Failed to destroy cuBLAS handle: %d", status);
    }
    g_handle = nullptr;
  }
}

void cublas_tensor_op_nn(half *A, half *B, half *C,  size_t M, size_t N, size_t K) {
	// 2. 定义标量系数
  static half alpha = 1.0;
  static half beta = 0.0;

  if (g_handle == nullptr) {
    init_cublas_handle();
  }

  cublasGemmEx(g_handle, 
               CUBLAS_OP_N, 
               CUBLAS_OP_N, 
               N, M, K, 
               &alpha, 
               B, CUDA_R_16F, N, 
               A, CUDA_R_16F, K, 
               &beta,  
               C, CUDA_R_16F, N, 
               CUBLAS_COMPUTE_16F,
               CUBLAS_GEMM_DEFAULT_TENSOR_OP);
}

int main() {
    // 矩阵维度定义 (m, n, k)
    // A: m x k, B: k x n, C: m x n
    const int m = 2, n = 2, k = 3;
    float h_A[m * k] = {1, 2, 3, 4, 5, 6};
    float h_B[k * n] = {7, 8, 9, 10, 11, 12};
    float h_C[m * n] = {0};

    float *d_A, *d_B, *d_C;
    cudaMalloc(&d_A, m * k * sizeof(float));
    cudaMalloc(&d_B, k * n * sizeof(float));
    cudaMalloc(&d_C, m * n * sizeof(float));

    cudaMemcpy(d_A, h_A, m * k * sizeof(float), cudaMemcpyHostToDevice);
    cudaMemcpy(d_B, h_B, k * n * sizeof(float), cudaMemcpyHostToDevice);

    // 1. 创建 cuBLAS 句柄
    cublasHandle_t handle;
    cublasCreate(&handle);
  
  	cublas_tensor_op_nn(d_A, d_B, d_C, M, N, K);
   

    // 将结果拷回主机
    cudaMemcpy(h_C, d_C, m * n * sizeof(float), cudaMemcpyDeviceToHost);

    // 释放资源
    cublasDestroy(handle);
    cudaFree(d_A); cudaFree(d_B); cudaFree(d_C);

    return 0;

```

### cutlass

NVIDIA 的 **CUTLASS** (CUDA Templates for Linear Algebra Subroutines) 是一个非常强大的高效 CUDA C++ 模板库，它将矩阵乘法（GEMM）分解为可重用的分层组件。

要理解 CUTLASS 的实现，最直观的方法是看它的 **Device API**，这是最推荐的集成方式。

------

**1. CUTLASS GEMM 核心代码示例**

下面是一个典型的使用 CUTLASS 3.x 风格（或经典 2.x 接口）实现 $C = \alpha (A \times B) + \beta C$ 的简化代码框架。



```C++
#include <iostream>
#include <cutlass/cutlass.h>
#include <cutlass/gemm/device/gemm.h>

// 定义矩阵的维度
int M = 1024;
int N = 1024;
int K = 1024;

// 1. 定义数据类型和布局
using ElementA = float; 
using ElementB = float; 
using ElementC = float; 
using ElementAccumulator = float; // 累加器类型

using LayoutA = cutlass::layout::RowMajor;
using LayoutB = cutlass::layout::RowMajor;
using LayoutC = cutlass::layout::RowMajor;

// 2. 实例化 CUTLASS GEMM 内核
using Gemm = cutlass::gemm::device::Gemm<
    ElementA, LayoutA,
    ElementB, LayoutB,
    ElementC, LayoutC,
    ElementAccumulator,
    cutlass::arch::OpClassSimt,    // 算术算子类（Simt 或 TensorOp）
    cutlass::arch::Sm80            // 目标架构（如 Ampere SM80）
>;

int main() {
    // 准备数据（省略 CUDA 分配和初始化过程...）
    float alpha = 1.0f;
    float beta = 0.0f;

    // 3. 构造 GEMM 参数对象
    typename Gemm::Arguments args(
        {M, N, K},          // 问题维度
        {ptrA, lda},        // A 指针及主维步长
        {ptrB, ldb},        // B 指针及主维步长
        {ptrC, ldc},        // C 指针及主维步长（源）
        {ptrC, ldc},        // C 指针及主维步长（结果）
        {alpha, beta}       // 标量系数
    );

    // 4. 初始化并执行
    Gemm gemm_op;
    cutlass::Status status = gemm_op(args);

    if (status != cutlass::Status::kSuccess) {
        return -1;
    }
    return 0;
}
```

------

**2. 代码深度解析**

CUTLASS 的核心哲学是**分层分块（Hierarchical Tiling）**。

**A. 类型定义（Templates）**

与传统的 cuBLAS 不同，CUTLASS 使用模板参数来决定硬件指令。

- **`OpClassSimt` vs `OpClassTensorOp`**：如果你有 Tensor Core（如 RTX 30/40 系列），改为 `TensorOp` 会让性能提升数倍。
- **`Sm80`**：指定编译器针对特定的显卡架构优化。

**B. 分层分块机制 (Tiling)**

CUTLASS 能够高效的原因在于它将巨大的矩阵乘法拆解成了多个层级：

1. **Device Level (Grid)**：将整个输出矩阵 $C$ 划分为多个 **Threadblock Tiles**。
2. **Threadblock Level**：每个线程块负责一块数据。它从 Global Memory 异步加载数据到 **Shared Memory**。
3. **Warp Level**：线程块内部，数据被进一步划分为 **Warp Tiles**，每个 Warp（32个线程）从 Shared Memory 取数。
4. **Instruction Level**：最后由具体的 CUDA 指令（如 `mma.sync`）在寄存器层面完成计算。

**C. 软件流水线 (Software Pipelining)**

CUTLASS 大量使用了 `cp.async`（异步拷贝指令）。这意味着当计算单元正在处理第 $n$ 块数据时，内存拷贝单元已经在后台从显存搬运第 $n+1$ 块数据到共享内存了。这种**计算与访存的重叠**是达到峰值性能的关键。

**D. Epilogue（收尾操作）**

在矩阵相乘完成后，CUTLASS 提供了一个专门的 `Epilogue` 阶段。在这里处理 $\alpha$ 和 $\beta$ 的缩放，或者应用激活函数（如 ReLU）。因为这部分操作是访存密集型的，集成在 GEMM 内核中可以避免额外的显存读写。

------

**3. 为什么选择 CUTLASS 而不是 cuBLAS？**

- **灵活性**：你可以轻松修改内核，在 GEMM 之后直接接一个 `LeakyRelu` 或 `BiasAdd`。
- **深度集成**：由于是纯头文件库，你可以将它直接编译进你的工程，不需要额外的动态库依赖。
- **融合算子 (Fusion)**：在 AI 推理中，GEMM 后接各种算子是常态，CUTLASS 是实现高性能 **Kernel Fusion** 的首选工具。

### cute

**1. CuTe 核心逻辑实现**

```c++
 * * 模板参数说明：
 * T: 数据类型 (如 half_t)
 * BM, BN, BK: Thread Block 层次的矩阵分块大小
 * kStage: 流水线阶段数 (用于异步拷贝隐藏延迟)
 * TiledMMA: 描述 MMA 指令如何组织的类型
 * G2SCopyA/B: Global 到 Shared Memory 的拷贝算子
 * SmemLayoutA/B/C: Shared Memory 的数据布局
 * S2RCopyAtomA/B: Shared 到 Register 的拷贝原子
 * R2SCopyAtomC/S2GCopyC: 寄存器到 Shared 继而到 Global 的拷贝逻辑
 * BlockSwizzle: 是否启用线程块重排以优化 L2 缓存命中率
 */
template <
      typename T, 
      int BM, 
      int BN, 
      int BK, 
      int kStage, 
      typename TiledMMA, 
      typename G2SCopyA, 
      typename G2SCopyB,
      typename SmemLayoutA, 
      typename SmemLayoutB, 
      typename SmemLayoutC,
      typename S2RCopyAtomA, 
      typename S2RCopyAtomB,
      typename R2SCopyAtomC, 
      typename S2GCopyAtomC, 
      typename S2GCopyC,
      const bool BlockSwizzle>
__global__ void hgemm_mma_stages_block_swizzle_tn_cute_kernel(
  T *Aptr, T *Bptr, T *Dptr, int m, int n, int k) {
  using namespace cute;

  // --- 1. 初始化 Shared Memory ---
  extern __shared__ T shm_data[];
  T *Ashm = shm_data;
  T *Bshm = shm_data + cute::cosize(SmemLayoutA{}); // B 的偏移取决于 A 的大小

  // --- 2. 线程块索引计算 (含 Block Swizzle) ---
  int idx = threadIdx.x;
  // 如果启用 BlockSwizzle，则通过某种映射改变 blockIdx 的读取顺序，优化缓存
  int ix = ((int) BlockSwizzle) * blockIdx.z * gridDim.x + blockIdx.x;
  int iy = blockIdx.y;

  // 边界检查
  if (iy * BM >= m || ix * BN >= n) return;

  // --- 3. 定义全局内存 Tensor ---
  // A: (M, K), B: (N, K), D: (M, N)
  Tensor A = make_tensor(make_gmem_ptr(Aptr), make_shape(m, k), make_stride(k, Int<1>{}));
  Tensor B = make_tensor(make_gmem_ptr(Bptr), make_shape(n, k), make_stride(k, Int<1>{}));
  Tensor D = make_tensor(make_gmem_ptr(Dptr), make_shape(m, n), make_stride(n, Int<1>{}));

  // 切割全局内存：获取当前线程块负责处理的局部 Tile
  Tensor gA = local_tile(A, make_tile(Int<BM>{}, Int<BK>{}), make_coord(iy, _)); // (BM, BK, num_tile_k)
  Tensor gB = local_tile(B, make_tile(Int<BN>{}, Int<BK>{}), make_coord(ix, _)); // (BN, BK, num_tile_k)
  Tensor gD = local_tile(D, make_tile(Int<BM>{}, Int<BN>{}), make_coord(iy, ix));// (BM, BN) 

  // --- 4. 定义共享内存 Tensor ---
  auto sA = make_tensor(make_smem_ptr(Ashm), SmemLayoutA{}); // (BM, BK, kStage)
  auto sB = make_tensor(make_smem_ptr(Bshm), SmemLayoutB{}); // (BN, BK, kStage)

  // --- 5. MMA 与 寄存器分片 (Partitioning) ---
  TiledMMA tiled_mma;
  auto thr_mma = tiled_mma.get_slice(threadIdx.x);
  
  // 划分结果矩阵 D 的寄存器
  auto tCgD = thr_mma.partition_C(gD);                // (MMA, MMA_M, MMA_N)
  auto tCrD = thr_mma.partition_fragment_C(gD);       // (MMA, MMA_M, MMA_N)
  clear(tCrD); // 累加器清零

  // --- 6. 数据搬运划分 (Global -> Shared) ---
  G2SCopyA g2s_tiled_copy_a;
  auto g2s_thr_copy_a = g2s_tiled_copy_a.get_slice(idx);
  auto tAgA_copy = g2s_thr_copy_a.partition_S(gA);     // 源：Global Tile
  auto tAsA_copy = g2s_thr_copy_a.partition_D(sA);     // 目的：Shared Tile

  G2SCopyB g2s_tiled_copy_b;
  auto g2s_thr_copy_b = g2s_tiled_copy_b.get_slice(idx);
  auto tBgB_copy = g2s_thr_copy_b.partition_S(gB);
  auto tBsB_copy = g2s_thr_copy_b.partition_D(sB);

  // --- 7. 数据搬运划分 (Shared -> Register) ---
  auto s2r_tiled_copy_a = make_tiled_copy_A(S2RCopyAtomA{}, tiled_mma);
  auto s2r_thr_copy_a = s2r_tiled_copy_a.get_slice(idx);
  auto tAsA = s2r_thr_copy_a.partition_S(sA);          // 源：Shared
  auto tCrA = thr_mma.partition_fragment_A(gA(_, _, 0));
  auto tCrA_view = s2r_thr_copy_a.retile_D(tCrA);      // 转换视图以匹配 Copy 算子

  auto s2r_tiled_copy_b = make_tiled_copy_B(S2RCopyAtomB{}, tiled_mma);
  auto s2r_thr_copy_b = s2r_tiled_copy_b.get_slice(idx);
  auto tBsB = s2r_thr_copy_b.partition_S(sB);
  auto tCrB = thr_mma.partition_fragment_B(gB(_, _, 0));
  auto tCrB_view = s2r_thr_copy_b.retile_D(tCrB);

  /* --- 8. 预取 (PREFETCH) 阶段 --- */
  int itile_to_read = 0;
  int ismem_read = 0;
  int ismem_write = 0;

  // 填充流水线：预先加载 kStage - 1 个 Tile 到共享内存
  #pragma unroll
  for (int istage = 0; istage < kStage - 1; ++istage) {
    cute::copy(g2s_tiled_copy_a, tAgA_copy(_, _, _, istage), tAsA_copy(_, _, _, istage));
    cute::copy(g2s_tiled_copy_b, tBgB_copy(_, _, _, istage), tBsB_copy(_, _, _, istage));
    cp_async_fence(); // 提交异步拷贝任务

    ++itile_to_read;
    ++ismem_write;
  }

  // 等待第一批数据到达 Shared Memory (保留 kStage-2 个任务在队列中)
  cp_async_wait<kStage - 2>();
  __syncthreads();

  // 加载第一个计算所需的 Tile 到寄存器 (Shared -> Register)
  int ik = 0;
  cute::copy(s2r_tiled_copy_a, tAsA(_, _, ik, ismem_read), tCrA_view(_, _, ik)); 
  cute::copy(s2r_tiled_copy_b, tBsB(_, _, ik, ismem_read), tCrB_view(_, _, ik));

  // --- 9. 主循环 (流水线计算) ---
  int ntile = k / BK;
  #pragma unroll 1
  for (int itile = 0; itile < ntile; ++itile) {
    int nk = size<2>(tCrA); // K 维度上的微块数

    #pragma unroll
    for (int ik = 0; ik < nk; ++ik) {
      int ik_next = (ik + 1) % nk;
      
      // 如果当前是微块的最后一步，准备切换到下一个共享内存阶段
      if (ik == nk - 1) {
        cp_async_wait<kStage - 2>(); // 等待下一个阶段的异步拷贝完成
        __syncthreads();
        ismem_read = (ismem_read + 1) % kStage;
      }
      
      // 预取下一个微块的数据到寄存器 (双缓冲/流水线思想)
      cute::copy(s2r_tiled_copy_a, tAsA(_, _, ik_next, ismem_read), tCrA_view(_, _, ik_next));
      cute::copy(s2r_tiled_copy_b, tBsB(_, _, ik_next, ismem_read), tCrB_view(_, _, ik_next));

      // 在计算开始时，发射下一轮从 Global 到 Shared 的异步拷贝
      if (ik == 0) {
        if (itile_to_read < ntile) {
          cute::copy(g2s_tiled_copy_a, tAgA_copy(_, _, _, itile_to_read), tAsA_copy(_, _, _, ismem_write));
          cute::copy(g2s_tiled_copy_b, tBgB_copy(_, _, _, itile_to_read), tBsB_copy(_, _, _, ismem_write));
          ++itile_to_read;
          ismem_write = (ismem_write + 1) % kStage;
        }
        cp_async_fence();
      }

      // 执行核心计算：Tensor Core MMA 指令
      cute::gemm(tiled_mma, tCrD, tCrA(_, _, ik), tCrB(_, _, ik), tCrD);
    }  // end ik
  } // end itile

  // --- 10. 结果写回 (Epilogue) ---
  // 使用 A 的 Shared Memory 空间作为临时缓冲区（Scratchpad）
  auto sC = make_tensor(sA(_, _, ismem_read).data(), SmemLayoutC{});

  auto r2s_tiled_copy_c = make_tiled_copy_C(R2SCopyAtomC{}, tiled_mma);
  auto r2s_thr_copy_c = r2s_tiled_copy_c.get_slice(idx);
  auto tCrC_r2s = r2s_thr_copy_c.retile_S(tCrD);   // 寄存器源
  auto tCsC_r2s = r2s_thr_copy_c.partition_D(sC);  // Shared 目的

  S2GCopyC s2g_tiled_copy_c;
  auto s2g_thr_copy_c = s2g_tiled_copy_c.get_thread_slice(idx);
  auto tCsC_s2g = s2g_thr_copy_c.partition_S(sC);  // Shared 源
  auto tCgC_s2g = s2g_thr_copy_c.partition_D(gD);  // Global 目的

  // 合并维度以简化循环逻辑
  auto tCgC_s2gx = group_modes<1, 3>(tCgC_s2g); 
  auto tCrC_r2sx = group_modes<1, 3>(tCrC_r2s); 

  int step = size<3>(tCsC_r2s); 
  #pragma unroll
  for (int i = 0; i < size<1>(tCrC_r2sx); i += step) {
    // 第一步：寄存器 -> Shared Memory
    #pragma unroll
    for (int j = 0; j < step; ++j) {
      auto t = make_tensor_like<T>(tCrC_r2sx(_, i + j));
      cute::copy(tCrC_r2sx(_, i + j), t); // 处理数据类型转换

      cute::copy(r2s_tiled_copy_c, t, tCsC_r2s(_, 0, 0, j));
    }
    __syncthreads(); // 确保所有线程都写完 Shared Memory

    // 第二步：Shared Memory -> Global Memory
    #pragma unroll
    for (int j = 0; j < step; ++j) {
      cute::copy(s2g_tiled_copy_c, tCsC_s2g(_, 0, 0, j), tCgC_s2gx(_, i + j));
    }
    __syncthreads(); // 确保写回完成，以便下一轮复用 Shared Memory
  } 
}
```

------

**2. 代码深度解析**

**A. Layout（布局）的魔力**

在传统的 CUDA 代码中，你需要手动计算索引：`A[row * ld + col]`。

而在 CuTe 中，**Layout** 将物理存储与逻辑坐标分离。

- `dA` 包含了矩阵的形状（Shape）和步长（Stride）。
- 当你调用 `mA(m_idx, k)` 时，CuTe 内部会根据 `dA` 自动计算出对应的内存偏移量。这使得代码可以无缝切换 **Row-Major**（行优先）或 **Col-Major**（列优先），而无需修改核心算法逻辑。

**B. Tensor 抽象**

`make_tensor` 函数将一个基础指针（如 `float*`）与一个 `Layout` 结合。

- **Global Tensor**: 指向 GPU 显存。

- **Shared Tensor**: 指向共享内存。

- **Register Tensor**: 指向寄存器。

  这种统一的抽象允许开发者使用相同的语法访问不同层级的存储。

**C. 层次化切分 (Tiling & Partitioning)**

虽然上面的代码为了易读性使用了简单的循环，但 CuTe 的真正威力在于 **Tiling**（分块）。在高性能场景下，我们会这样操作：

1. **Threadblock Level**: 使用 `local_tile` 将大矩阵切成小块放入 Shared Memory。
2. **Thread Level**: 使用 `thr_partition` 将 Shared Memory 的数据分发给各个线程的寄存器。

------

**3. 进阶：CuTe 的典型执行流**

一个高性能的 CuTe GEMM 通常遵循以下步骤：

| **步骤**          | **操作**                             | **CuTe 工具**                      |
| ----------------- | ------------------------------------ | ---------------------------------- |
| **1. 定义布局**   | 定义 M, N, K 的形状及步长            | `make_layout`, `make_shape`        |
| **2. 内存拷贝**   | 将数据从 Global 搬运到 Shared (Smem) | `copy()`, `make_tiled_copy()`      |
| **3. 寄存器加载** | 从 Smem 加载到寄存器 (RF)            | `thr_partition`, `local_partition` |
| **4. 计算**       | 在寄存器层面执行乘加运算             | `gemm()`, `make_tiled_mma()`       |

------

**4. 为什么选择 CuTe 而不是原生 CUDA？**

1. **类型安全与解耦**：你不需要在代码里写死索引计算公式。
2. **自动优化**：CuTe 能够自动利用硬件指令（如 NVIDIA Tensor Cores）。
3. **高度可组合**：你可以轻松地把一个 Layout 嵌套在另一个 Layout 之中，实现复杂的重排（Permutation）。









 ### cute详细解释

```c++
using SmemLayoutAtom = decltype(composition(
            Swizzle<3, 3, 3> {}, make_layout(make_shape(Int<8> {}, Int<BK> {}), make_stride(Int<BK> {}, Int<1> {}))));
```

这块代码的核心目的只有四个字：**消除冲突**。

 1. 拆解 `composition` (组合)

在 CuTe 中，`composition(A, B)` 的作用是**“把函数 B 的输出作为函数 A 的输入”**。

* **Layout 本质上是一个函数**：输入一个“逻辑坐标”（比如第几行、第几列），输出一个“内存偏移量”。
* **组合的过程**：
1. `make_layout` 定义了一个最基础的存储方式（比如按行存或按列存）。
2. `Swizzle` 定义了一个“地址洗牌”规则。
3. `composition` 把这两个结合起来。当你访问 `(row, col)` 时，它先计算出原始地址，然后立刻用 `Swizzle` 把这个地址“洗”一下，最后才真正去读内存。



  2. **`Swizzle<3, 3, 3>` 做了什么：**
     它通过位运算（通常是 `XOR` 异或）对地址进行变换。

* **第一个 3 (B)**：表示参与变换的位宽（Base）。
* **第二个 3 (M)**：表示变换的周期（Mask）。
* **第三个 3 (S)**：表示左移的位数（Shift）。
简单来说，它把原本在内存里“排成直线”的数据，变成了“交叉分布”。这样当你按列读的时候，原本会撞车的线程被散开到了不同的 Bank 上。

 3. `make_layout` 里的参数

```cpp
make_layout(make_shape(Int<8> {}, Int<BK> {}),
            make_stride(Int<BK> {}, Int<1> {}))

```

这定义了一个 （假设 ）的微型矩阵：

* **Shape (8, 32)**：8 行 32 列。
* **Stride (32, 1)**：
* `1`：意味着相邻的列在内存里是连续的（行优先）。
* `32`：意味着跳到下一行需要跨过 32 个元素。



 4. 为什么用 `decltype`？

在 C++ 模板编程中，这些 Layout 的类型极其复杂（包含了嵌套的模板和静态常量）。

* 你不可能手写出这个类型的名字（可能长达几百个字符）。
* `decltype(...)` 的意思就是：**“编译器，你自己算一下后面这一坨表达式推导出来的类型是什么，然后把这个类型给 `SmemLayoutAtom` 变量。”**
* 这是一种**编译期静态绑定**，不产生任何运行时开销。

---

 总结一下这块代码在干嘛：

1. **定义基础结构**：先弄一个  的小格子。
2. **加入“洗牌”算法**：用 `Swizzle` 告诉编译器，读这些格子的时候地址要异或一下，躲开 Bank 冲突。
3. **封装类型**：用 `composition` 把格子结构和洗牌算法打包。
4. **定名**：用 `decltype` 拿到这个复杂的打包类型，起个名字叫 `SmemLayoutAtom`。

**一句话总结：它在定义一种“为了防止内存读写打架而专门设计的数据排列艺术”。**

---

```c++
using SmemLayoutA = decltype(tile_to_shape(SmemLayoutAtom {},
    make_shape(Int<BM> {}, Int<BK> {}, Int<KStage> {})));
using SmemLayoutB = decltype(tile_to_shape(SmemLayoutAtom {},
    make_shape(Int<BN> {}, Int<BK> {}, Int<KStage> {}))); // (m,n) -> smem_idx
```



如果说前面的 `SmemLayoutAtom` 是定义了一块“**这种花纹的地砖**”，那么这一行 `tile_to_shape` 就是在做“**铺地板**”的工作：把那一块块地砖整齐地铺满整个大房间。

我们来逐个剥开它的逻辑：

------

 1. `tile_to_shape` 是干什么的？

顾名思义，它的作用是 **“平铺到一个形状”**。

- **输入**：一个小的布局（Atom，即地砖）和一个大的目标形状（Shape，即房间大小）。
- **输出**：一个新的布局。这个新布局会把小的 Atom 在空间上不断重复，直到填满大的目标形状。

在这个例子中：

- 你的 Atom（地砖）是前面定义的那个带 `Swizzle` 的 $8 \times 32$ 布局。
- 你的目标形状是 `(BM, BK, KStage)`，也就是 $128 \times 32 \times 2$（假设 $BM=128, BK=32, KStage=2$）。

 2. 为什么要带上 `KStage`？（关键：多级缓存）

注意这里的目标形状是 **3D** 的：`(M, K, Stage)`。

- **M 和 K**：代表一个数据块（Tile）的大小。
- **Stage**：这是 **Double Buffering（双缓冲）** 或 **Multi-stage（多级流水）** 的核心。
  - 在内存里，它会开辟 `KStage` 份空间。
  - 当计算单元（Tensor Core）正在处理第 `0` 份数据时，搬运单元（cp.async）可以同时往第 `1` 份空间里写下一轮的数据。
  - `tile_to_shape` 确保了每一层（每一个 Stage）都继承了 Atom 定义的那种“防 Bank 冲突”的 Swizzle 模式。

 3. 这里的维度是如何对应的？

让我们看看 `tile_to_shape` 内部发生的“拼图”过程：

- **第一维 (M)**：目标是 128。Atom 只有 8。所以它会在 M 方向上重复 $128 / 8 = 16$ 次。
- **第二维 (K)**：目标是 32。Atom 已经是 32。所以 K 方向不做重复，刚好匹配。
- **第三维 (Stage)**：目标是 `KStage`（比如 2）。Atom 只有 2D 属性。`tile_to_shape` 会自动扩展出一个新的维度。

最终，`SmemLayoutA` 就变成了一个逻辑上是 $128 \times 32 \times 2$ 的大张量。

 4. 为什么又是 `decltype`？

还是那个原因：**类型太复杂了**。

`tile_to_shape` 返回的不是一个简单的数组指针，而是一个嵌套了多层模板的 CuTe 对象。这个对象内部记录了：

1. 如何从 `(m, k, stage)` 计算出内存偏移。
2. 如何应用 Swizzle 变换。
3. 如何处理维度的步长（Stride）。

通过 `decltype`，你让编译器在编译的时候就把这套“地址计算公式”全部推导出来并固定死。这样在内核运行时，所有的地址跳转都是**零开销**的（因为都是常数位移和简单的位运算）。

------

 形象化理解：

想象你有一张带防伪水印（Swizzle）的小纸条（Atom）。

`tile_to_shape` 就像是一台复印机，它把这张小纸条复印了很多份，整齐地贴在一个大箱子（BM x BK）里。

而且它还准备了两个这样的箱子（KStage=2），并把这两个箱子摞在一起。

**结果就是**：你现在拥有了一个“**两层楼高的、每一寸墙面都带防冲突水印的、逻辑连续的大仓库**”。

------

 总结

这行代码完成了 **Shared Memory 的逻辑建模**：

- 它告诉线程块：你的 A 矩阵缓冲区总共有多大。
- 它定义了如何把 $M, K, Stage$ 三个坐标映射到那块物理 Shared Memory 指针上。
- 它保证了无论你在哪一层、哪个位置读数据，都符合最初设计的 **Swizzle 规则**。

**这就是 CuTe 的强大之处**：你不需要自己去算 `ptr + m*32 + k + stage*128*32` 这种容易出错的偏移量，你只需要定义好这个“形状”和“地砖”，剩下的地址映射全部由 `SmemLayoutA` 自动搞定。

你觉得这个“铺地砖”的过程解释清楚了吗？如果还是觉得抽象，我们可以直接看一眼 Kernel 里的 `make_tensor` 是怎么把这个 Layout 套到那个 `shm_data` 指针上的。



```c++
using mma_op = SM80_16x8x16_F16F16F16F16_TN;
using mma_traits = MMA_Traits<mma_op>;
using mma_atom = MMA_Atom<mma_traits>;
```

这三行代码是进入 **Tensor Core（张量核心）** 硬件底层的心脏。它们不是在写软件算法，而是在**描述硬件电路的规格**。



------

 1. `mma_op = SM80_16x8x16_F16F16F16F16_TN`

这一行是在**选定具体的硬件指令**。

- **SM80**: 指的是 NVIDIA **Ampere（安培）架构**（如 A100, RTX 30系列）。每个架构的 Tensor Core 指令都不一样。
- **16x8x16**: 这是这个硬件“零件”的**物理尺寸**。它意味着这条指令一次性能吞掉一个 $16 \times 16$ 的 A 矩阵块和一个 $16 \times 8$ 的 B 矩阵块，吐出一个 $16 \times 8$ 的结果 C。
- **F16F16F16F16**: 描述的是**数据精度**。
  - 前两个 F16：输入矩阵 A 和 B 是 `half` 精度。
  - 后两个 F16：累加器和输出矩阵 C 也是 `half` 精度。
- **TN**: 描述的是**内存布局（Transpose / Normal）**。
  - **T**: 矩阵 A 是转置的。
  - **N**: 矩阵 B 是正常的。
  - 这对底层非常重要，因为硬件电路必须知道数据是按行喂进来还是按列喂进来。

------

 2. `mma_traits = MMA_Traits<mma_op>`

这一行是在**提取说明书**。

`MMA_Traits` 就像是针对上面那个具体指令的“特性提取器”。它会告诉编译器：

- 这条指令需要多少个线程参与？（对于 `SM80_16x8x16`，通常是一个 Warp，即 32 个线程）。
- 每个线程需要持有矩阵 A、B、C 的哪些部分？（线程和数据之间的映射关系）。
- 每个线程需要多少个寄存器？

**大白话**：指令只是个名字，`Traits` 把它背后的参数（如：需要多少人干活、每个人拿多少东西）给查了出来。

------

 3. `mma_atom = MMA_Atom<mma_traits>`

这一行是**封装成一个“原子”算子**。

这是 CuTe 最核心的抽象概念之一。**Atom（原子）** 的意思是：**它是不可分割的最小计算单元**。

- 在 CuTe 里，你不能直接指挥 Tensor Core，你得先定义一个“原子”。
- 这个 `mma_atom` 包含了上面所有的信息（硬件指令 + 线程映射 + 数据特性）。
- **为什么要封装？** 因为后面的 `make_tiled_mma` 会像“复制粘贴”一样，把这个小小的 `mma_atom` 在空间上平铺开，形成一个能处理大矩阵（如 $128 \times 256$）的巨大计算网格。

------

 整体串起来理解：

1. **`mma_op`**：告诉编译器：“我要用 A100 里面那个 $16 \times 8 \times 16$ 的半精度硬件加法器。”
2. **`mma_traits`**：编译器回答：“行，我知道了。这个加法器需要 32 个人（Warp）一组，每个人得手里拿几块数据。”
3. **`mma_atom`**：你把这一切打包：“好，这就是我的**最小作战单元**，后面我就按这个标准来分工。”

 为什么这里不涉及具体的计算逻辑？

因为 CuTe 采用的是 **声明式编程**。你在这里先定义好“武器（Atom）”的规格，后面只需要一句话 `cute::gemm(...)`，CuTe 就会根据你这里定义的 `mma_atom` 自动推导出最优的汇编代码。

**一句话总结：这三行是在给硬件“选型”并拉取它的“出厂参数”。**

这下能听懂了吗？如果还是觉得这些名字起得很晦涩，我可以给你对比一下 SM75（Turing 架构）的定义有什么不同，你一眼就能看出它们的规律。

接下来想了解 `make_tiled_mma` 是怎么把这个“原子”变成“大网”的吗？那是代码中控制 **Warp 布局** 的核心。





```c++
static constexpr int kMmaEURepeatM = 2; // MMA repeat 2 times across M
static constexpr int kMmaEURepeatN = 2; // MMA repeat 2 times across N
static constexpr int kMmaEURepeatK = 1; // MMA no repeat across K

using mma_atom_shape = mma_traits::Shape_MNK; // M,N,K 16,8,16
static constexpr int kMmaPM = 1 * kMmaEURepeatM * get<0>(mma_atom_shape {}); // 1*2*16=32
static constexpr int kMmaPN = 2 * kMmaEURepeatN * get<1>(mma_atom_shape {}); // 2*2*8 =32
static constexpr int kMmaPK = 1 * kMmaEURepeatK * get<2>(mma_atom_shape {}); // 1*1*16=16
// TiledMMA, more threads, MMAThrLayout(2,2,1), 4 MMA = 4 warps = 32x4 threads.
using MMA_EU_RepeatT = decltype(make_layout(make_shape(
    Int<kMmaEURepeatM> {}, Int<kMmaEURepeatN> {}, Int<kMmaEURepeatK> {})));
// TiledMMA, more values, Permutations(32,32,16)
using MMA_P_T = Tile<Int<kMmaPM>, Int<kMmaPN>, Int<kMmaPK>>;
using MMA = decltype(make_tiled_mma(mma_atom {}, MMA_EU_RepeatT {}, MMA_P_T {}));
```

这一段代码是整个 GEMM 设计中最具“策略性”的部分。如果说之前的 `mma_atom` 是选定了一把**手枪**，那么这段代码就是在定义如何把这些手枪组成一个**火力加强排**。

它决定了线程块（Thread Block）内部的 **128个线程（4个 Warp）是如何排兵布阵的**。

------

 1. `EURepeat` (Execution Unit Repeat)：兵力部署

C++

```
static constexpr int kMmaEURepeatM = 2; 
static constexpr int kMmaEURepeatN = 2; 
static constexpr int kMmaEURepeatK = 1; 
```

这里的 **EU（执行单元）** 指的就是 **Warp**（32个线程组成的最小调度单位）。

- **M=2, N=2**：意味着在一个线程块内部，我们安排了 $2 \times 2 = 4$ 个 Warp。
- **物理布局**：你可以想象成一个田字格，每个格子对应一个 Warp。
- **总线程数**：$4 \text{ Warps} \times 32 \text{ threads} = 128$ 线程。这就是为什么后面的 `block(size(MMA{}))` 算出来是 128。

 2. 计算“逻辑地砖”的大小 (`kMmaPM`, `kMmaPN` 等)

这一步是在算：这 4 个 Warp 凑在一起，一口气能吃掉多大的矩阵块？

C++

```
using mma_atom_shape = mma_traits::Shape_MNK; // 硬件原子是 16x8x16
```

- **`kMmaPM` (32)**：$1 \times 2 \times 16 = 32$。表示 M 方向覆盖 32 行。
- **`kMmaPN` (32)**：$2 \times 2 \times 8 = 32$。表示 N 方向覆盖 32 列。
  - *注意：这里代码里乘了一个额外的系数 2，是为了匹配更高的计算吞吐量。*
- **`kMmaPK` (16)**：$1 \times 1 \times 16 = 16$。K 方向覆盖 16 个元素。

**结论**：这个“排”一次齐射，可以完成一个 $32 \times 32 \times 16$ 的小型矩阵乘法。

 3. `MMA_EU_RepeatT`：线程布局映射

C++

```
using MMA_EU_RepeatT = decltype(make_layout(make_shape(
    Int<kMmaEURepeatM> {}, Int<kMmaEURepeatN> {}, Int<kMmaEURepeatK> {})));
```

这行代码正式告诉 CuTe：**“请把线程索引映射到 M=2, N=2 的 Warp 阵列上。”**

CuTe 会自动帮你计算出：`threadIdx.x` 从 0 到 31 归 Warp0，32 到 63 归 Warp1，以此类推，并让它们分别负责结果矩阵 C 的不同象限。

 4. `MMA_P_T` (Permutations/Permuted Tile)：数据重排策略

C++

```
using MMA_P_T = Tile<Int<kMmaPM>, Int<kMmaPN>, Int<kMmaPK>>;
```

这定义了在这个 TiledMMA 层级上，数据是如何“排列组合”的。它指定了最终合成的大块数据的逻辑形状（$32 \times 32 \times 16$）。

 5. `make_tiled_mma`：大功告成

C++

```
using MMA = decltype(make_tiled_mma(mma_atom {}, MMA_EU_RepeatT {}, MMA_P_T {}));
```

这是最终的“母体”对象。它把 **硬件原子指令**、**线程怎么分工**、**数据怎么切块** 全部揉在了一起。

**它的威力在于：**

一旦你定义了 `MMA`，在 Kernel 里的 `thr_mma.partition_C(gD)` 这一行，CuTe 就会根据这里的定义，**自动计算出**第 `i` 个线程应该去全局内存的哪个位置读数，应该把结果存在哪个寄存器里。

------

 总结：这几行代码解决了什么问题？

如果没有 CuTe，你需要手动写类似于这样的代码：

```
int warp_id = threadIdx.x / 32;
int lane_id = threadIdx.x % 32;
int my_row = (warp_id / 2) * 16 + ...;
int my_col = (warp_id % 2) * 8 + ...;
```

**这极其容易出错且性能低下。**

**现在的逻辑是：**

1. **定义士兵** (`mma_atom`)。
2. **定义阵型** (`MMA_EU_RepeatT` 为 2x2)。
3. **定义目标** (`MMA_P_T` 为 32x32)。
4. **生成指挥官** (`MMA`)。

**一句话总结：这几行代码是在通过“声明”而非“计算”，来规划 128 个线程如何最高效地协同完成一个 $32 \times 32$ 的计算分片。**

这一段理解后，整个 Host 端的配置就基本通了。你现在对这个“线程布局（Thread Layout）”的概念感觉如何？想不想看看 Kernel 里面是怎么调用这个 `MMA` 来“切”数据的？





```c++
using g2s_copy_op = SM80_CP_ASYNC_CACHEGLOBAL<cute::uint128_t>;
using g2s_copy_traits = Copy_Traits<g2s_copy_op>;
using g2s_copy_atom = Copy_Atom<g2s_copy_traits, T>;
using G2SCopyA = decltype(make_tiled_copy(g2s_copy_atom {},
    make_layout(make_shape(Int<32> {}, Int<4> {}), // Thr layout 32x4 k-major
        make_stride(Int<4> {}, Int<1> {})),
    make_layout(make_shape(Int<1> {}, Int<8> {})))); // Val layout 1x8
using G2SCopyB = G2SCopyA;
using s2r_copy_op = SM75_U32x4_LDSM_N;
using s2r_copy_traits = Copy_Traits<s2r_copy_op>;
using s2r_copy_atom = Copy_Atom<s2r_copy_traits, T>;
using S2RCopyAtomA = s2r_copy_atom;
using S2RCopyAtomB = s2r_copy_atom;
```



这一段代码是在定义**搬运工（Copy Operators）**。在高性能矩阵乘法中，数据搬运的效率和计算同样重要。这里定义了两条搬运流水线：一条是从**全局内存到共享内存（G2S）**，另一条是从**共享内存到寄存器（S2R）**。

我们用“物流系统”的逻辑来拆解：

---

 1. 全局内存 → 共享内存 (G2S)：高效异步运输

```cpp
using g2s_copy_op = SM80_CP_ASYNC_CACHEGLOBAL<cute::uint128_t>;

```

* **`SM80_CP_ASYNC`**: 又是 Ampere 架构（SM80）的专属武器。`cp.async` 指令允许硬件直接在内存之间搬数据，**不经过寄存器**。这就好比你在装修，材料直接从卡车（Global）卸到门口（Shared），不用工人手提肩扛搬进屋（Register）。
* **`uint128_t`**: 指定搬运的“货车”载重。一次搬 128 位（16字节），这是 NVIDIA 硬件搬运效率最高的位宽。

```cpp
using G2SCopyA = decltype(make_tiled_copy(g2s_copy_atom {},
    make_layout(make_shape(Int<32> {}, Int<4> {}), // 线程布局 32x4
                make_stride(Int<4> {}, Int<1> {})),
    make_layout(make_shape(Int<1> {}, Int<8> {})))); // 每个线程拿 8 个元素

```

这里是在定义**搬运工的阵型**：

* **线程布局 (32, 4)**：128 个线程被排成 32 行 4 列。
* **值布局 (1, 8)**：每个线程负责一行中的 8 个连续元素（）。
* **总规模**：32行  (4列  每人搬8个) = 。刚好对应了一个  块中的一部分（或者是为了对齐访问步长）。

---

 2. 共享内存 → 寄存器 (S2R)：喂给计算单元

```cpp
using s2r_copy_op = SM75_U32x4_LDSM_N;

```

* **`LDSM` (Load Shared Memory)**: 这是从共享内存加载数据到寄存器的顶级指令。
* **为什么用它？** 因为 Tensor Core 计算时，对寄存器里数据的排列顺序有非常古怪的要求。`LDSM` 指令在读取共享内存时，会自动完成一些“数据变换”，让数据读进寄存器后直接就是 Tensor Core 喜欢的姿势。
* **`SM75`**: 虽然整体是 SM80 的代码，但 LDSM 指令是从 Turing（SM75）架构开始引入的，所以这里沿用了这个名字。

---

 3. `using G2SCopyB = G2SCopyA;`

* 这行很简单：矩阵 B 的搬运逻辑和 A 完全一样。因为在 TN 布局下，A 和 B 在  维度上的访问特征是对称的。

---

 4. 重点梳理：这两组搬运有什么区别？

| 阶段    | 指令 (Atom) | 起点 → 终点       | 核心优势                                             |
| ------- | ----------- | ----------------- | ---------------------------------------------------- |
| **G2S** | `cp.async`  | Global → Shared   | **异步**。发完指令立刻去干别的，不需要等待搬运完成。 |
| **S2R** | `LDSM`      | Shared → Register | **高效布局**。自动处理 Tensor Core 所需的数据排布。  |

---

 总结

这部分代码定义了 **“数据怎么挪动”**。

* **G2SCopy** 保证了数据能从遥远的显存，源源不断且不占 CPU 周期地流入线程块的仓库（Shared Memory）。
* **S2RCopy** 保证了仓库里的数据，能以最符合硬件要求的格式，精准地喂给 Tensor Core。

**一句话总结：这几行是在定义“物流搬运标准”，确保搬运不会成为计算的绊脚石。**

到这里，所有的“静态定义”都结束了，接下来代码就进入了计算 Grid、分配显存并正式启动 Kernel 的部分。你对这套“海陆空联运”的物流系统还有疑问吗？比如为什么一定要用 128bit？





```c++

using SmemLayoutAtomC = decltype(composition(
        Swizzle<3, 3, 3> {},
        make_layout(make_shape(Int<kMmaPM> {}, Int<kMmaPN> {}), // 32*32
            make_stride(Int<kMmaPN> {}, Int<1> {}))));
using SmemLayoutC = decltype(tile_to_shape(
    SmemLayoutAtomC {},
    make_shape(Int<kMmaPM> {}, Int<kMmaPN> {}, Int<kSmemLayoutCBatch> {})));

static_assert(
    size<0>(SmemLayoutA {}) * size<1>(SmemLayoutA {}) >= size(SmemLayoutC {}),
    "C shared memory request is large than A's one pipe");
using R2SCopyAtomC = Copy_Atom<UniversalCopy<int>, T>;

using S2GCopyAtomC = Copy_Atom<UniversalCopy<cute::uint128_t>, T>;
using S2GCopyC = decltype(make_tiled_copy(
    S2GCopyAtomC {},
    make_layout(make_shape(Int<32> {}, Int<4> {}),
        make_stride(Int<4> {}, Int<1> {})),
    make_layout(make_shape(Int<1> {}, Int<8> {}))));
```

这一段代码定义的是 **Epilogue（收尾阶段）** 的布局和搬运方式。简单来说，就是矩阵乘法算完了，结果 $C$ 还在寄存器里，我们需要把它顺顺当当地写回到全局内存（Global Memory）里去。

这里有个非常精妙的**内存复用**设计，我们来逐行透视：

------

 1. `SmemLayoutAtomC` 与 `SmemLayoutC`：临时中转站

C++

```
using SmemLayoutAtomC = decltype(composition(
    Swizzle<3, 3, 3> {},
    make_layout(make_shape(Int<kMmaPM> {}, Int<kMmaPN> {}), // 32*32
                make_stride(Int<kMmaPN> {}, Int<1> {}))));
```

- **逻辑形状**：定义了一个 $32 \times 32$ 的布局（由前面的 `kMmaPM` 和 `kMmaPN` 算出）。
- **Swizzle**：同样应用了 `Swizzle<3, 3, 3>`。
  - **为什么结果写回也要 Swizzle？** 因为寄存器写回共享内存，以及随后从共享内存读出到全局内存，如果不 Swizzle，同样会因为 Bank 冲突导致写回变慢。
- **`tile_to_shape`**：将其扩展为 3D 形状 `(32, 32, Batch)`。这里的 `Batch` 通常是为了平铺多个计算结果块。

 2. `static_assert`：内存复用的“安全锁”

C++

```
static_assert(
    size<0>(SmemLayoutA {}) * size<1>(SmemLayoutA {}) >= size(SmemLayoutC {}),
    "C shared memory request is large than A's one pipe");
```

这是**整段代码最聪明的地方**。

- **背景**：Shared Memory 非常昂贵且有限。
- **策略**：当计算结束进入收尾阶段时，A 矩阵的输入数据已经没用了。因此，我们**把计算结果 C 存放在原本属于 A 的共享内存空间里**。
- **这行的意义**：在编译阶段检查，“仓库 A”的大小是否放得下“货物 C”。如果 C 太大放不下，编译器直接报错，防止运行时的内存溢出。

 3. `R2SCopyAtomC`：从寄存器到共享内存

C++

```
using R2SCopyAtomC = Copy_Atom<UniversalCopy<int>, T>;
```

- **`UniversalCopy<int>`**：这是一种通用的拷贝方式。
- **作用**：把存储在每个线程寄存器里的累加值（Accumulator）搬运到共享内存的“中转站”里。

 4. `S2GCopyC`：从共享内存到全局内存

C++

```
using S2GCopyAtomC = Copy_Atom<UniversalCopy<cute::uint128_t>, T>;
using S2GCopyC = decltype(make_tiled_copy(
    S2GCopyAtomC {},
    make_layout(make_shape(Int<32> {}, Int<4> {}), // 线程阵型 32x4
                make_stride(Int<4> {}, Int<1> {})),
    make_layout(make_shape(Int<1> {}, Int<8> {})))); // 128-bit 齐射
```

- **阵型**：同样采用了 128 个线程（32x4）协同作战。
- **128-bit (`uint128_t`)**：写回全局内存时，依然坚持使用 128-bit 宽度。
- **为什么不直接从寄存器写回全局内存？**
  - 寄存器里的数据分布通常是支离破碎的（为了配合 Tensor Core）。
  - **Shared Memory 起到了“打包装箱”的作用**。先在 Shared Memory 里排好序，然后用 128-bit 指令整块地写回显存，这样能触发显存的**合并访问（Coalesced Access）**，跑满带宽。

------

 总结：收尾阶段的物流链

1. **寄存器 (R)**：存放着刚算好的结果，但位置很乱。
2. **R2S**：把乱序的结果塞进共享内存（复用 A 的空间），顺便用 `Swizzle` 避开 Bank 冲突。
3. **S2G**：数据在共享内存里排整齐后，用 `uint128_t` 大卡车（S2GCopyC）把数据一口气拉回显存。

**一句话总结：这几行是在设计“如何利用旧仓库（A 的 Smem）来快速、有序地把战利品（结果 C）运回总部”。**



```c++
template <
    typename T,
    int BM,
    int BN,
    int BK,
    int kStage,
    typename TiledMMA,
    typename G2SCopyA,
    typename G2SCopyB,
    typename SmemLayoutA,
    typename SmemLayoutB,
    typename SmemLayoutC,
    typename S2RCopyAtomA,
    typename S2RCopyAtomB,
    typename R2SCopyAtomC,
    typename S2GCopyAtomC,
    typename S2GCopyC,
    const bool BlockSwizzle>
__global__ void hgemm_mma_stages_block_swizzle_tn_cute_kernel(
    T* Aptr, T* Bptr, T* Dptr, int m, int n, int k)
{
    using namespace cute;
    // Initilize shared memory
    extern __shared__ T shm_data[];

    T* Ashm = shm_data;
    T* Bshm = shm_data + cute::cosize(SmemLayoutA {});

    // Initilize thread block
    int idx = threadIdx.x;
    // BlockSwizzle 0/1 control use block swizzle or not.
    int ix = ((int)BlockSwizzle) * blockIdx.z * gridDim.x + blockIdx.x;
    int iy = blockIdx.y;

    if (iy * BM >= m || ix * BN >= n)
        return;

    // use Tensor notation to represent device pointer + dimension
    Tensor A = make_tensor(make_gmem_ptr(Aptr), make_shape(m, k), make_stride(k, Int<1> {}));
    Tensor B = make_tensor(make_gmem_ptr(Bptr), make_shape(n, k), make_stride(k, Int<1> {}));
    Tensor D = make_tensor(make_gmem_ptr(Dptr), make_shape(m, n), make_stride(n, Int<1> {}));

    // slice the tensor to small one which is used for current thread block.
    Tensor gA = local_tile(A, make_tile(Int<BM> {}, Int<BK> {}), make_coord(iy, _)); // (BM, BK, num_tile_k)
    Tensor gB = local_tile(B, make_tile(Int<BN> {}, Int<BK> {}), make_coord(ix, _)); // (BN, BK, num_tile_k)
    Tensor gD = local_tile(D, make_tile(Int<BM> {}, Int<BN> {}), make_coord(iy, ix)); // (BM, BN)

    // shared memory
    auto sA = make_tensor(make_smem_ptr(Ashm), SmemLayoutA {}); // (BM, BK, kStage)
    auto sB = make_tensor(make_smem_ptr(Bshm), SmemLayoutB {}); // (BN, BK, kStage)

    // dispatch TileA/TileB/TileC mma tensor into thread fragment via partition
    TiledMMA tiled_mma;
    auto thr_mma = tiled_mma.get_slice(threadIdx.x);
    auto tCgD = thr_mma.partition_C(gD); // (MMA,MMA_M, MMA_N)

    auto tCrA = thr_mma.partition_fragment_A(gA(_, _, 0)); // (MMA, MMA_M, MMA_K)
    auto tCrB = thr_mma.partition_fragment_B(gB(_, _, 0)); // (MMA, MMA_N, MMA_K)
    auto tCrD = thr_mma.partition_fragment_C(gD); // (MMA, MMA_M, MMA_N)
    clear(tCrD);

    // from global memory to shared memory
    G2SCopyA g2s_tiled_copy_a;
    auto g2s_thr_copy_a = g2s_tiled_copy_a.get_slice(idx);
    auto tAgA_copy = g2s_thr_copy_a.partition_S(gA); // (CPY, CPY_M, CPY_K, num_tile_k)
    auto tAsA_copy = g2s_thr_copy_a.partition_D(sA); // (CPY, CPY_M, CPY_K, kStage)
    G2SCopyB g2s_tiled_copy_b;
    auto g2s_thr_copy_b = g2s_tiled_copy_b.get_slice(idx);
    auto tBgB_copy = g2s_thr_copy_b.partition_S(gB); // (CPY, CPY_N, CPY_K, num_tile_k)
    auto tBsB_copy = g2s_thr_copy_b.partition_D(sB); // (CPY, CPY_N, CPY_K, kStage)

    // from shared memory to register, use tiled_mma to generate tiled_copy
    auto s2r_tiled_copy_a = make_tiled_copy_A(S2RCopyAtomA {}, tiled_mma);
    auto s2r_thr_copy_a = s2r_tiled_copy_a.get_slice(idx);
    auto tAsA = s2r_thr_copy_a.partition_S(sA); // (CPY, CPY_M, CPY_K, kStage)
    auto tCrA_view = s2r_thr_copy_a.retile_D(tCrA); // (CPY, CPY_M, CPY_K)

    auto s2r_tiled_copy_b = make_tiled_copy_B(S2RCopyAtomB {}, tiled_mma);
    auto s2r_thr_copy_b = s2r_tiled_copy_b.get_slice(idx);
    auto tBsB = s2r_thr_copy_b.partition_S(sB); // (CPY, CPY_N, CPY_K, kStage)
    auto tCrB_view = s2r_thr_copy_b.retile_D(tCrB); // (CPY, CPY_N, CPY_K)

    /* PREFETCH */
    // submit kStage - 1 tile
    // gmem -> shm
    int itile_to_read = 0;
    int ismem_read = 0;
    int ismem_write = 0;

#pragma unroll
    for (int istage = 0; istage < kStage - 1; ++istage) {
        cute::copy(g2s_tiled_copy_a, tAgA_copy(_, _, _, istage),
            tAsA_copy(_, _, _, istage));
        cute::copy(g2s_tiled_copy_b, tBgB_copy(_, _, _, istage),
            tBsB_copy(_, _, _, istage));
        cp_async_fence();

        ++itile_to_read;
        ++ismem_write;
    }

    // wait one submitted gmem->smem done
    cp_async_wait<kStage - 2>();
    __syncthreads();

    int ik = 0;
    // smem -> reg
    // tAsA: (CPY, CPY_M, CPY_K, kStage) tCrA_view: (CPY, CPY_M, CPY_K)
    cute::copy(s2r_tiled_copy_a, tAsA(_, _, ik, ismem_read), tCrA_view(_, _, ik));
    cute::copy(s2r_tiled_copy_b, tBsB(_, _, ik, ismem_read), tCrB_view(_, _, ik));

    // loop over k: i. load tile, ii. mma
    int ntile = k / BK;
#pragma unroll 1
    for (int itile = 0; itile < ntile; ++itile) {
        int nk = size<2>(tCrA); // (MMA, MMA_M, MMA_K)

#pragma unroll
        for (int ik = 0; ik < nk; ++ik) {
            int ik_next = (ik + 1) % nk;

            if (ik == nk - 1) {
                cp_async_wait<kStage - 2>();
                __syncthreads();

                ismem_read = (ismem_read + 1) % kStage;
            }

            // shm -> reg s[itile][ik + 1] -> r[ik + 1]
            // tAsA: (CPY, CPY_M, CPY_K, kStage), tCrA_view: (CPY, CPY_M, CPY_K)
            cute::copy(s2r_tiled_copy_a, tAsA(_, _, ik_next, ismem_read),
                tCrA_view(_, _, ik_next));
            // tBsB: (CPY, CPY_M, CPY_K, kStage), tCrB_view: (CPY, CPY_M, CPY_K)
            cute::copy(s2r_tiled_copy_b, tBsB(_, _, ik_next, ismem_read),
                tCrB_view(_, _, ik_next));

            if (ik == 0) {
                if (itile_to_read < ntile) {
                    cute::copy(g2s_tiled_copy_a, tAgA_copy(_, _, _, itile_to_read),
                        tAsA_copy(_, _, _, ismem_write));
                    cute::copy(g2s_tiled_copy_b, tBgB_copy(_, _, _, itile_to_read),
                        tBsB_copy(_, _, _, ismem_write));
                    ++itile_to_read;
                    ismem_write = (ismem_write + 1) % kStage;
                }

                cp_async_fence();
            }

            cute::gemm(tiled_mma, tCrD, tCrA(_, _, ik), tCrB(_, _, ik), tCrD);
        } // for ik
    }

    // use less shared memory as a scratchpad tile to use large wide instuction
    // Dreg -> shm -> reg -> global
    auto sC = make_tensor(sA(_, _, ismem_read).data(), SmemLayoutC {});

    auto r2s_tiled_copy_c = make_tiled_copy_C(R2SCopyAtomC {}, tiled_mma);
    auto r2s_thr_copy_c = r2s_tiled_copy_c.get_slice(idx);
    auto tCrC_r2s = r2s_thr_copy_c.retile_S(tCrD); // (CPY, CPY_M, CPY_N)
    auto tCsC_r2s = r2s_thr_copy_c.partition_D(sC); // (CPY, _1, _1, pipe)

    S2GCopyC s2g_tiled_copy_c;
    auto s2g_thr_copy_c = s2g_tiled_copy_c.get_thread_slice(idx);
    auto tCsC_s2g = s2g_thr_copy_c.partition_S(sC); // (CPY, _1, _1, pipe)
    auto tCgC_s2g = s2g_thr_copy_c.partition_D(gD); // (CPY, CPY_M, CPY_N)

    auto tCgC_s2gx = group_modes<1, 3>(tCgC_s2g); // (CPY_, CPY_MN)
    auto tCrC_r2sx = group_modes<1, 3>(tCrC_r2s); // (CPY_, CPY_MN)

    int step = size<3>(tCsC_r2s); // pipe
#pragma unroll
    for (int i = 0; i < size<1>(tCrC_r2sx); i += step) {
// reg -> shm
#pragma unroll
        for (int j = 0; j < step; ++j) {
            // we add a temp tensor to cope with accumulator and output data type
            // difference
            auto t = make_tensor_like<T>(tCrC_r2sx(_, i + j));
            cute::copy(tCrC_r2sx(_, i + j), t);

            cute::copy(r2s_tiled_copy_c, t, tCsC_r2s(_, 0, 0, j));
        }
        __syncthreads();

#pragma unroll
        // shm -> global
        for (int j = 0; j < step; ++j) {
            cute::copy(s2g_tiled_copy_c, tCsC_s2g(_, 0, 0, j), tCgC_s2gx(_, i + j));
        }
        __syncthreads();
    } // end for
}
```

终于到了最硬核的部分。这个 **Kernel 函数**是高性能 GEMM 的“战场实体”。它采用了 **Software Pipelining (软件流水线)** 技术，实现了“边搬运、边计算”。

为了让你彻底看懂，我将内核分为四个阶段：**初始化、数据切片、主循环流水线、收尾写回**。

------

 1. 初始化与指针映射



```C++
extern __shared__ T shm_data[];
T* Ashm = shm_data;
T* Bshm = shm_data + cute::cosize(SmemLayoutA {});
```

- **共享内存切割**：在物理地址上，`Ashm` 是起点，`Bshm` 紧随其后。`cosize` 确保了 A 的空间不会被 B 覆盖。


```C++
int ix = ((int)BlockSwizzle) * blockIdx.z * gridDim.x + blockIdx.x;
int iy = blockIdx.y;
```

- **逻辑坐标计算**：根据是否开启 `BlockSwizzle` 计算当前线程块负责矩阵的哪个 Tile。`iy` 对应 M 轴，`ix` 对应 N 轴。

------

 2. 张量切片：从全局到局部 (Partitioning)

这是 CuTe 的精髓——**逻辑拆解**。

 全局视角 (Global Memory)


```C++
Tensor A = make_tensor(make_gmem_ptr(Aptr), make_shape(m, k), make_stride(k, Int<1> {}));
// ... 
Tensor gA = local_tile(A, make_tile(Int<BM> {}, Int<BK> {}), make_coord(iy, _)); 
```

- `make_tensor`：把显存指针封装成带有行列逻辑的张量。
- `local_tile`：从巨大的 A 矩阵中，切出当前 Block 负责的那一条“长廊”（$BM \times K$）。

 线程视角 (Thread Partition)

```C++
auto thr_mma = tiled_mma.get_slice(threadIdx.x);
auto tCrA = thr_mma.partition_fragment_A(gA(_, _, 0)); // 寄存器碎片
```

- **`partition_fragment_A`**：这步非常关键！它在**寄存器**里给当前线程分配了位置。每个线程只持有矩阵的一小部分碎片（Fragment），这一步确定了“我这个线程该拿哪几个数”。

------

 3. 软件流水线 (The Pipelining Mainloop)

这部分代码在处理一个核心矛盾：**显存太慢，计算太快**。

 预取阶段 (Prefetch)

```C++
#pragma unroll
for (int istage = 0; istage < kStage - 1; ++istage) {
    cute::copy(g2s_tiled_copy_a, tAgA_copy(_, _, _, istage), tAsA_copy(_, _, _, istage));
    cp_async_fence(); // 提交异步搬运指令
}
```

- 在正式开始计算前，先提前把前 `kStage-1` 份数据从全局内存拉到共享内存。`cp_async` 像发快递一样，发完就走，不阻塞。

 核心循环 (Main Loop)

```C++
for (int itile = 0; itile < ntile; ++itile) {
    for (int ik = 0; ik < nk; ++ik) {
        // 1. 等待数据到位
        if (ik == nk - 1) {
            cp_async_wait<kStage - 2>(); // 确保最老的那批货到了
            __syncthreads();
        }
        
        // 2. 从 Shared Memory 搬到 Register (S2R)
        cute::copy(s2r_tiled_copy_a, tAsA(_, _, ik_next, ismem_read), tCrA_view(_, _, ik_next));
        
        // 3. 计算 (MMA)
        cute::gemm(tiled_mma, tCrD, tCrA(_, _, ik), tCrB(_, _, ik), tCrD);
    }
}
```

- **计算与搬运重叠**：当 Tensor Core 在算第 `itile` 个块时，`cp_async` 已经在路上传输第 `itile + kStage` 个块了。
- **双指针轮转**：`ismem_read` 和 `ismem_write` 像传送带一样循环，保证读写不会撞车。

------

 4. 收尾写回 (Epilogue)

算完后，结果 `tCrD` 散落在 128 个线程的寄存器里。

```C++
auto sC = make_tensor(sA(_, _, ismem_read).data(), SmemLayoutC {});
```

- **空间复用**：`sC` 直接借用了 `sA` 的地址，省下了宝贵的共享内存。



```C++
#pragma unroll
for (int i = 0; i < size<1>(tCrC_r2sx); i += step) {
    // A. 寄存器 -> 共享内存 (R2S)
    cute::copy(tCrC_r2sx(_, i + j), t);
    cute::copy(r2s_tiled_copy_c, t, tCsC_r2s(_, 0, 0, j));
    __syncthreads();

    // B. 共享内存 -> 全局内存 (S2G)
    cute::copy(s2g_tiled_copy_c, tCsC_s2g(_, 0, 0, j), tCgC_s2gx(_, i + j));
    __syncthreads();
}
```

- **分批写回**：因为共享内存空间有限（复用的 A 空间可能放不下整个 C Tile），所以采用了 `step` 分批写回。
- **数据重整**：寄存器里的数据布局不连续，直接写显存很慢。先写进 Shared Memory 凑整齐，再用 `uint128` (128位) 大卡车拉回显存，效率最高。

------

 关键逻辑总结：

1. **异步化**：用 `cp_async` 让搬运不占计算时间。
2. **分级缓冲**：`Global -> Shared -> Register` 三级跳转。
3. **坐标解耦**：CuTe 的 `partition` 让你不需要手算复杂的 `threadIdx` 偏移。
4. **资源极限利用**：通过 `static_assert` 检查和 `data()` 指针复用，把 Shared Memory 压榨到极致。

**一句话总结：这个 Kernel 像一个高效的流水线工厂，原材料（A, B）不断进场，工人们（Thread）在工位上小块加工，最后成品（C）打包出场，全程没有一个人在闲着等。**
